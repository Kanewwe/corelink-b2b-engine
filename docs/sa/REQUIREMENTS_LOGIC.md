## 1. 探勘引擎 (Lead Mining Engine)

### 1.1 採集模式 (Scraper Modes)
系統目前支援三種主要探勘邏輯，均受 `180s Timeout` 與 `Auto-Retry` 保護：

| 模式 | 核心邏輯 | 關鍵組件 |
| :--- | :--- | :--- |
| **黃頁模式 (YellowPages)** | 模擬瀏覽器爬取 Google / Bing Search 結果並提取結構化數據。 | `direct_scraper.py` |
| **製造商模式 (Manufacturer)** | 針對 Thomasnet 等特定 B2B 目錄進行深度挖掘。 | `manufacturer_miner.py` |
| **免費掃描 (Free Hunter)** | 透過 `free_email_hunter.py` 進行網域暴力掃描與 Email 驗證。 | `free_email_hunter.py` |

### 1.2 全域池與私有池同步 (Pool Sync Logic)
為節省重複採集成本，系統實施兩層名單機制：
- **Global Pool (`public.global_leads`)**: 跨租戶共享的原始資料。
- **Private Pool (`{schema}.leads`)**: 租戶專屬名單，包含私有標籤與投遞狀態。
- **同步觸發**: 當 User 搜尋關鍵字時，引擎優先從 Global Pool 撈取，若無則啟動外部採集並同步寫入兩端。

## 3. 投遞與自動化 (Campaign & Outreach)

### 3.1 SMTP 資源分配 (System Settings Hub)
- **集中管理**: `system_settings` 表儲存所有 API Keys (Hunter, ScraperAPI, OpenAI) 與 SMTP 設定。
- **隔離邏輯**: 會員只能使用其所屬 `vendor_id` 配置或全域預設配置。

### 3.2 投遞狀態追蹤
- **發送鏈結**: `Campaign` → `Job` → `EmailLog`。
- **即時反饋**: 支援 `Sent`, `Opened`, `Clicked`, `Replied` 四種狀態回傳（透過 `email_tracker.py`）。

## 4. 系統架構基礎 (Architecture)

### 4.1 多租戶隔離 (Multi-Tenancy)
- **實作方式**: PostgreSQL Schema-based (例如 `uat.users`, `public.global_leads`)。
- **切換機制**: 在 FastAPI Middleware 根據 Session 中的 `vendor_id` 或資料庫連線字串動態設定 `search_path`。

### 4.2 任務控制 (Task Control)
- **背景任務**: 使用 `APScheduler` 定時掃描並執行待處理的 Campaign Job。
- **超時機制**: 爬蟲邏輯內嵌 `asyncio.wait_for(timeout=180)`。
